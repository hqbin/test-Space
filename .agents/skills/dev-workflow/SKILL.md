# 通用开发工作流

> 任何项目的开发任务都必须遵循此五阶段闭环流程。
> 被调用时，Agent 必须严格按阶段执行，不可跳过审查和测试。

---

## 概述

```
┌─────────────────────────────────────────────────────────┐
│                  ① 理解需求 (Understand)                  │
│   └─ 先读项目文档，再分析需求                              │
├─────────────────────────────────────────────────────────┤
│                  ② 制定计划 (Plan)                        │
│   └─ 拆解步骤，列出文件清单                                │
├─────────────────────────────────────────────────────────┤
│                  ③ 执行 (Execute)                        │
│   └─ 按计划编码，遵循项目规范                              │
├─────────────────────────────────────────────────────────┤
│                  ④ 审查 (Review)                         │
│   └─ 自检质量，同步更新文档                                │
├─────────────────────────────────────────────────────────┤
│                  ⑤ 测试验证 (Test)                        │
│   └─ 类型检查 → 构建 → 功能验证                           │
└─────────────────────────────────────────────────────────┘
```

**核心原则**：
- 每个阶段完成后才能进入下一阶段
- 不允许跳过"审查"或"测试"阶段
- 开发前必须先读项目文档，开发完成后必须同步更新文档

## 技能自动路由

开始任务时，先判断任务类型，自动调用对应的专业技能：

| 任务类型 | 判断条件 | 自动调用 |
|---------|---------|---------|
| **前端修改** | Vue/React 组件、CSS、布局、UI 交互、动效 | 调用 `frontend-design` 或对应前端 skill |
| **Bug 修复** | 功能异常、崩溃、错误、测试失败 | 调用 `systematic-debugging` 或对应调试 skill |
| **UI/UX 设计** | 视觉设计、配色、字体、图标、设计系统 | 调用 `ui-ux-pro-max` 或对应设计 skill |
| **其他开发任务** | 后端、数据库、脚本、配置、文档等 | 直接执行本工作流 |

> 调用方式：在"理解需求"阶段识别出任务类型后，立即用 Skill 工具调用对应技能。多类型任务则依次调用。

## 阶段一：理解需求

### 1.1 先读文档

**开发前必须先阅读项目文档**，了解项目结构和规范：

- 查找并阅读项目文档（`README.md`、`DEVELOPMENT.md`、`CONTRIBUTING.md` 等）
- 理解技术栈和架构
- 理解编码规范和 UI/UX 约定
- 理解数据持久化方案
- 理解项目目录结构

如果没有找到项目文档，至少阅读 `package.json` / `Cargo.toml` 了解依赖和脚本。

### 1.2 分析需求

- [ ] 确认任务的具体目标和范围
- [ ] 识别涉及的功能模块
- [ ] 判断影响范围：前端 / 后端 / 数据库 / 配置
- [ ] 如需求不明确，提出具体澄清问题
- [ ] 如涉及破坏性操作（删数据、改安全配置等），标注风险

**完成标志**：能清晰陈述"要做什么"和"要改什么"。

---

## 阶段二：制定计划

- [ ] 任务已分解为 1 到 N 个具体步骤
- [ ] 列出每个步骤涉及的文件路径
- [ ] 标注需要同步更新项目文档的步骤
- [ ] 标注需要数据库迁移的步骤
- [ ] 定义了每个步骤的完成标准
- [ ] 对跨文件改动，标注了影响范围
- [ ] （可选）对复杂任务，创建 `task_plan.md` 持久化计划

### 计划格式示例

```markdown
### Step 1: 修改数据层
- 文件: `src/services/database.ts`
- 内容: 新增 XXX 表迁移，添加 CRUD 函数
- 完成标准: CRUD 函数签名完整，类型定义正确

### Step 2: 新增页面组件
- 文件: `src/views/xxx/XXXPage.vue`
- 内容: 新建页面，包含列表和表单
- 完成标准: 页面可渲染，数据正确加载

### Step 3: 更新文档
- 文件: `DEVELOPMENT.md`
- 内容: 新增功能描述
- 完成标准: 文档匹配实际实现
```

> **计划制定后，必须等待用户确认才能进入执行阶段。**（但对本用户的日常任务无需询问，直接执行。）

---

## 阶段三：执行

### 编码原则

1. **最小改动，最大兼容** — 优先增量调整，避免大规模重构
2. **修改公共模块前评估影响** — 改公共模块时检查所有调用方
3. **删改前全局搜索** — 删除/重命名导出、路由、CSS 类、API 字段前，先搜索确认无其他引用
4. **注释"为什么"而非"是什么"** — 解释业务决策、边界处理

### 开发中记录变更

开发过程中自动记录以下信息，用于最终更新文档：

- 新增/修改的文件路径
- 改动的核心逻辑说明
- 新增的数据库表/字段、API 端点、命令
- 新增的依赖
- 任何需要注意的架构决策或兼容性说明

---

## 阶段四：审查

### 自检清单

#### 代码正确性
- [ ] 改动是否符合计划预期的范围
- [ ] 是否有遗漏的文件未修改
- [ ] 新代码是否遵循项目的类型安全要求
- [ ] 是否有潜在的空指针/undefined 访问风险
- [ ] 错误处理是否完整

#### 兼容性
- [ ] 修改公共模块时，所有调用方是否正常工作
- [ ] 删除/重命名前已全局搜索确认
- [ ] 数据库迁移是否可安全重入

#### 一致性
- [ ] 新代码与项目现有编码风格一致
- [ ] UI 元素符合项目的设计系统规范
- [ ] 未引入新的样式/交互模式（须复用现有规范）

#### 文档同步（**必须**）
- [ ] 项目文档是否需要同步更新？（功能变更必须更新）
- [ ] 如新增 API/命令，是否需要更新对应文档
- [ ] 如修改表结构，是否需要更新备份导入导出逻辑
- [ ] 如新增依赖，是否需要更新依赖清单

#### 安全
- [ ] 没有硬编码的敏感信息（token、密钥）
- [ ] 用户输入已做净化处理
- [ ] 没有引入新的注入风险

#### 清理
- [ ] 所有调试代码（console.log、dbg! 等）已删除
- [ ] 临时文件/目录已清理
- [ ] 注释不含 TODO 但没有对应追踪

---

## 阶段五：测试验证

### 必须执行的验证（按顺序）

#### 1. 类型检查
```bash
# 根据项目语言执行对应的类型检查
npm run typecheck    # TypeScript
cargo check          # Rust
# 或其他
```
> 必须通过，零错误。

#### 2. 构建验证
```bash
# 根据项目类型执行构建
npm run build        # Node/Vite/Webpack
cargo build          # Rust
# 或其他
```
> 必须通过，零错误。

#### 3. 功能冒烟验证

根据修改内容选择以下验证项：

- **页面/UI 渲染**：新页面/组件能否正常渲染
- **数据读写**：增删改查是否正常，数据持久化是否正确
- **API/命令**：新增的接口能否正常调用和返回
- **UI 交互**：按钮点击、弹窗、表单提交是否正常
- **路由**：路由跳转、参数传递是否正常
- **快捷键**：是否影响现有快捷键绑定
- **构建产物**：打包后功能是否正常

### 验证结果格式

```
## 验证报告

### 类型检查 ✅ / ❌
- 结果: [通过 / 失败]

### 生产构建 ✅ / ❌
- 结果: [通过 / 失败]

### 功能验证 ✅ / ❌
- [ ] 页面渲染正常
- [ ] 数据读写正常
- [ ] API/命令正常
- [ ] UI 交互正常
- [ ] 无回归问题
```

## 特殊情况处理

### 仅文档修改
可简化流程：理解需求 → 执行 → 审查。跳过"制定计划"和"测试验证"。

### 紧急修复
可在用户明确要求下跳过"制定计划"阶段，但"审查"和"测试验证"不可跳过。

### 新项目初始化
- "理解需求"阶段须额外确认项目类型和技术选型
- 首次执行后应生成项目文档（README.md / DEVELOPMENT.md）
- 项目文档模板可在后续迭代中逐步完善

## 项目文档管理

### 开发文档应包含的内容

建议项目文档（如 `DEVELOPMENT.md`）包含以下章节：

1. **项目概述** — 产品定位、技术栈
2. **技术架构** — 层级、依赖、目录结构
3. **开发环境** — 环境要求、安装步骤
4. **编码规范** — 命名、样式、组件规范
5. **数据持久化** — 数据库、文件存储方案
6. **API 参考** — Tauri 命令、REST API 等
7. **发布流程** — 版本号、打包、发布

### 变更记录

功能变更后应在项目文档末尾或单独的更新日志中记录：

```markdown
### Phase N — 功能名称

| 模块 | 说明 | 状态 |
|------|------|------|
| 模块名 | 具体改动 | ✅ |

**验证**：
- 构建: ✅ 通过
- 类型检查: ✅ 通过
```
