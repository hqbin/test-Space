# Test Space 开发工作流

> 这是 Test Space 项目的核心开发工作流技能。
> 被调用时，Agent 必须严格遵循下面的五阶段闭环流程。

---

## 概述

本技能定义了 Test Space 项目中 Agent 执行开发任务的 **标准操作流程**。任何时候处理开发任务（新增功能、修复 Bug、重构、优化），都必须按以下五阶段闭环执行：

```
┌─────────────────────────────────────────────────────────┐
│                  ① 理解需求 (Understand)                  │
├─────────────────────────────────────────────────────────┤
│                  ② 制定计划 (Plan)                        │
├─────────────────────────────────────────────────────────┤
│                  ③ 执行 (Execute)                        │
├─────────────────────────────────────────────────────────┤
│                  ④ 审查 (Review)                         │
├─────────────────────────────────────────────────────────┤
│                  ⑤ 测试验证 (Test)                        │
└─────────────────────────────────────────────────────────┘
```

**重要规则**：
- 每个阶段完成后才能进入下一阶段
- 不允许跳过"审查"或"测试"阶段直接输出结果
- 如用户明确要求跳过某阶段（如"不要测试"），须提醒风险并征得确认

## 技能自动路由

**每次开始任务时，必须先判断任务类型，自动调用对应的专业技能：**

| 任务类型 | 判断条件 | 自动调用 Skill |
|---------|---------|---------------|
| **前端修改** | 涉及 Vue 组件、CSS、布局、样式、UI 交互、动效 | 先调用 `frontend-design`，再执行本工作流 |
| **Bug 修复** | 功能异常、崩溃、错误、测试失败、意外行为 | 先调用 `systematic-debugging`，再执行本工作流 |
| **UI/UX 设计** | 视觉设计、配色、字体、图标、设计系统 | 先调用 `ui-ux-pro-max`，再执行本工作流 |
| **其他开发任务** | 后端、数据库、脚本、配置、文档等 | 直接执行本工作流 |

> **调用方式**：在"理解需求"阶段，识别出任务类型后，立即使用 `Skill` 工具调用对应的 skill，加载其指令后再继续执行五阶段流程。如果任务同时涉及多种类型（如修 Bug 需要改前端 UI），则依次调用多个 skill。**直接执行，不需要再问我确认。**

## 阶段一：理解需求

### 必须完成的检查

- [ ] 确认任务的具体目标和范围
- [ ] 识别涉及的功能模块（Device Space / Notes Space / API Space / Script Space / Settings / Rust 后端）
- [ ] 判断影响范围：前端（Vue）/ 后端（Rust）/ 数据库（SQLite）/ 配置（Tauri config）
- [ ] 如需求不明确，提出具体澄清问题
- [ ] 如涉及破坏性操作（删数据、改安全配置等），标注风险

### 常用参考文件

```
test-space/DEVELOPMENT.md    → 项目整体架构和功能说明
TestSpace_API.md            → 云端备份 API
.agents/AGENTS.md           → AGENT 规范（完整编码约定）
src/router/index.ts         → 路由结构
src/types/index.ts          → 类型定义
src/services/database.ts    → 数据库模式
```

### 输出

清晰地描述：
- **做什么**：一句话描述任务目标
- **改什么**：受影响的功能区域和文件
- **风险**：是否有破坏性操作

---

## 阶段二：制定计划

### 必须完成的检查

- [ ] 任务已分解为 1 到 N 个具体步骤
- [ ] 列出每个步骤涉及的文件路径（完整的绝对或相对路径）
- [ ] 标注需要同步更新 `DEVELOPMENT.md` 的步骤
- [ ] 标注需要数据库迁移的步骤
- [ ] 定义了每个步骤的完成标准
- [ ] 对跨文件改动，标注了影响范围（谁依赖被改的模块）
- [ ] （可选）对复杂任务，创建 `task_plan.md` 持久化计划

### 计划格式示例

```markdown
## 执行计划

### Step 1: 修改数据层
- 文件: `src/services/database.ts`
- 内容: 新增 XXX 表迁移，添加 CRUD 函数
- 完成标准: CRUD 函数签名完整，类型定义正确

### Step 2: 新增页面组件
- 文件: `src/views/xxx/XXXPage.vue`
- 内容: 新建页面，包含列表和表单
- 完成标准: 页面可渲染，数据正确加载

### Step 3: 注册路由
- 文件: `src/router/index.ts`
- 内容: 添加新路由
- 完成标准: 路由可访问

### Step 4: 更新文档
- 文件: `test-space/DEVELOPMENT.md`
- 内容: 新增功能描述
- 完成标准: 文档匹配实际实现
```

### 用户确认

> **计划制定后，必须等待用户确认才能进入执行阶段。**
> 特别是有破坏性操作、跨文件重构、或架构级变更时。

## 阶段三：执行

### 必须遵守的编码原则

1. **最小改动，最大兼容** — 优先增量调整，避免大规模重构
2. **修改公共模块前评估影响** — 改 composable、store、API 层、路由、布局组件时，检查所有调用方
3. **删改前全局搜索** — 删除/重命名导出、路由、CSS 类、API 字段前，`grep` 确认无其他引用
4. **注释"为什么"而非"是什么"** — 解释业务决策、边界处理，不要重复代码本身

### UI 规范速查

| 场景 | 正确做法 |
|------|----------|
| 按钮/可交互元素 | `class="glass-button"` — 禁用 `text-primary` |
| 鼠标悬停效果 | `class="glass-hover"` |
| 活跃状态 | `class="glass-active"` — 禁用 `bg-primary-container` |
| 弹窗遮罩 | `bg-black/30 backdrop-blur-sm` |
| 弹窗主体 | `glass-panel rounded-[2rem] bg-white/60` |
| 弹窗关闭按钮 | `glass-button p-1 rounded` + `close` 图标 |

### 数据持久化速查

| 场景 | 方案 |
|------|------|
| 跨会话持久化业务数据 | SQLite（`app_settings` 或专用表） |
| 临时/会话级状态 | `localStorage` |
| 键值对配置 | `getSetting('/setSetting` |
| 新增结构化数据 | 添加 `CREATE TABLE IF NOT EXISTS` 到 `migrateInternal()` |
| 新增表后的备份 | 同步更新 `AppBackup` / `exportAllData` / `importAllData` |

### 执行中的注意事项

- Rust 端：`#[cfg(target_os = "windows")]` 设置 `creation_flags(0x08000000)` 防止弹出控制台窗口
- Rust 端：耗时 ADB/IO 操作使用 `tokio::task::spawn_blocking`
- 前端：使用 `@/` 别名引用 `src/` 下的模块
- 数据库：`try/catch` 包裹 FTS5 相关代码（降级方案）
- 清理：**删除调试代码**（`console.log`、`dbg!`、临时注释）

## 阶段四：审查

### 自检清单

#### 代码正确性
- [ ] 改动是否符合计划预期的范围（没有做计划外的事）
- [ ] 是否有遗漏的文件未修改
- [ ] 新代码是否遵循 TypeScript/Rust 类型安全要求
- [ ] 是否有潜在的空指针/undefined 访问风险
- [ ] Rust 端错误处理是否完整（不 unwrap() 用户输入/外部数据）

#### 兼容性
- [ ] 修改公共模块时，所有调用方是否正常工作
- [ ] 删除/重命名前已全局搜索确认
- [ ] 数据库迁移是否可安全重入（`try/catch` 包裹 `ALTER TABLE`）

#### 一致性
- [ ] 新代码与项目现有编码风格一致
- [ ] UI 元素符合液态玻璃设计系统规范
- [ ] 未引入新的样式模式（必须复用现有 CSS 类）

#### 文档
- [ ] `DEVELOPMENT.md` 是否需要同步更新？（功能变更必须更新）
- [ ] 如新增 API/命令，是否需要更新 API 文档？
- [ ] 如修改表结构，是否需要更新备份导入导出逻辑？

#### 安全
- [ ] 没有硬编码的敏感信息（token、密钥）
- [ ] 用户输入已做净化处理（如 `DOMPurify`）
- [ ] 没有引入新的指令注入风险

#### 清理
- [ ] 所有 `console.log`、`dbg!`、调试代码已删除
- [ ] 临时的文件/目录已清理
- [ ] 注释不包含 TODO 但没有对应的 Issue/Plan

## 阶段五：测试验证

### 必须执行的验证（按顺序）

#### 1. 类型检查
```bash
cd test-space && npx vue-tsc --noEmit
```
> 必须通过，零错误。

#### 2. 生产构建
```bash
cd test-space && npm run build
```
> 必须通过，零错误。

#### 3. 功能冒烟验证

根据修改内容选择以下验证项（必须至少覆盖修改路径）：

- **页面渲染**：新页面/组件能否正常渲染，各 Tab 正常切换
- **数据读写**：增删改查是否正常，数据在重启后是否正确持久化
- **Rust 命令**：Tauri 命令能否正常调用和返回
- **UI 交互**：按钮点击、弹窗显示/关闭、表单提交是否正常
- **路由**：路由跳转、参数传递是否正常
- **键盘快捷键**：是否影响现有快捷键绑定（Ctrl+Tab、Ctrl+1~5）
- **单实例**：二次打开是否聚焦已有窗口
- **备份恢复**：如果需要，验证 export/import 流程

### 验证结果报告格式

```
## 验证报告

### 类型检查 ✅ / ❌
- 结果: [通过 / 失败]
- 错误数: [N]

### 生产构建 ✅ / ❌
- 结果: [通过 / 失败]

### 功能验证 ✅ / ❌
- [ ] 页面渲染正常
- [ ] 数据读写正常
- [ ] Rust 命令正常
- [ ] UI 交互正常
- [ ] 路由正常
- [ ] 无回归问题
```

## 特殊情况处理

### 修复已有 Bug
遵循相同五阶段流程，但在"理解需求"阶段要确认：
- Bug 复现步骤和条件
- 正常行为应该是什么
- 是否有现有测试可以覆盖

### 重构
- "制定计划"阶段必须详细列出重构前后对比
- "审查"阶段额外检查：功能等价性（重构不改变行为）
- "测试"阶段额外检查：所有涉及模块的完整冒烟测试

### 仅文档修改
- 如只修改文档（README、DEVELOPMENT.md 等），可简化流程：
  - 理解需求 → 执行 → 审查
  - 跳过"制定计划"和"测试验证"阶段
  - 但"审查"阶段仍然需要（检查格式、拼写、准确性）

### 紧急修复（如生产环境故障）
- 可在用户明确要求下跳过"制定计划"阶段
- 但"审查"和"测试验证"阶段不可跳过 — 紧急不等于粗糙
- 修复完成后补充计划文档以记录

## 与 AGENTS.md 的关系

- `.agents/AGENTS.md` 是项目级 AGENT **规范文档**，定义了所有 Agent 应遵循的全局规则和项目上下文
- 本 SKILL.md 是 **可执行的工作流技能**，将规范转化为具体的操作步骤和检查清单
- 两个文件配合使用：AGENTS.md 提供"为什么"和"规则"，SKILL.md 提供"怎么做"和"清单"
