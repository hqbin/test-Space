# Test Space — AGENT 规范文档

> 本文档定义了 AI 助手（Agent）在此项目中的工作规范和闭环流程。
> 所有 Agent 在执行任务时必须遵循本文档所述的阶段和准则。

---

## 一、项目背景

**Test Space** 是一款面向软件测试工程师的 Windows 桌面工作台，使用 **Tauri 2 + Vue 3 + Vite** 技术栈构建。它将测试用例设计、ADB 调试、串口调试、日志分析、脚本执行、API 代理/测试、知识沉淀等功能整合到一个桌面应用中。

### 技术栈概览

| 层级 | 技术选型 |
|------|----------|
| 桌面框架 | Tauri 2 (Rust) |
| 前端框架 | Vue 3 (Composition API + `<script setup>`) |
| 构建工具 | Vite 6 |
| 类型系统 | TypeScript |
| 状态管理 | Pinia |
| 路由 | Vue Router 4 |
| 样式 | TailwindCSS 3 + 液态玻璃设计系统 |
| 数据库 | SQLite (@tauri-apps/plugin-sql) |
| 原生层 | Rust（ADB、串口、脚本执行、MITM 代理） |
| 图标 | Material Symbols Outlined |
| 字体 | Inter |

> 详细依赖清单见项目根目录 `test-space/DEVELOPMENT.md`。

## 二、核心工作流（闭环五阶段）

每次执行任务必须按照以下五个阶段形成闭环，**每个阶段完成后方可进入下一阶段**：

```
┌─────────────────────────────────────────────────┐
│                  任务请求                         │
└─────────────────┬───────────────────────────────┘
                  ▼
┌─────────────────────────────────────────────────┐
│  ① 理解需求（Understand）                         │
│  ─ 分析请求，明确目标                              │
│  ─ 识别受影响模块                                  │
│  ─ 提出澄清问题（如有歧义）                        │
└─────────────────┬───────────────────────────────┘
                  ▼
┌─────────────────────────────────────────────────┐
│  ② 制定计划（Plan）                               │
│  ─ 拆解为可执行步骤                                │
│  ─ 列出需修改/创建的文件清单                        │
│  ─ 定义完成标准和验证方法                           │
│  ─ 确认计划后进入执行                              │
└─────────────────┬───────────────────────────────┘
                  ▼
┌─────────────────────────────────────────────────┐
│  ③ 执行（Execute）                                │
│  ─ 编写/修改代码                                  │
│  ─ 遵循项目编码规范                                │
│  ─ 最小改动、最大兼容                              │
│  ─ 标注关键逻辑注释                                │
└─────────────────┬───────────────────────────────┘
                  ▼
┌─────────────────────────────────────────────────┐
│  ④ 审查（Review）                                 │
│  ─ 自检所有变更                                    │
│  ─ 检查是否有破坏性变更                             │
│  ─ 验证编码规范和设计一致性                         │
│  ─ 确保文档同步更新                                │
└─────────────────┬───────────────────────────────┘
                  ▼
┌─────────────────────────────────────────────────┐
│  ⑤ 测试验证（Test）                                │
│  ─ 执行类型检查 (`vue-tsc --noEmit`)               │
│  ─ 执行构建 (`npm run build`)                     │
│  ─ 验证功能正确性                                  │
│  ─ 确认无回归问题                                  │
└─────────────────┬───────────────────────────────┘
                  ▼
┌─────────────────────────────────────────────────┐
│              任务完成 / 提交结果                    │
└─────────────────────────────────────────────────┘
```

### 2.1 第一阶段：理解需求

**目标**：在动手之前彻底理解用户的真实需求。

- 阅读并解析用户的请求
- 如需求不明确、存在歧义或缺少关键信息，主动提出澄清问题
- 识别请求涉及的功能模块（Device Space / Notes Space / API Space / Script Space / Settings）
- 判断影响范围：前端（Vue）、后端（Rust）、数据库（SQLite）、还是跨层改动
- 如有安全隐患或破坏性操作，提前指出并征求用户意见

**完成标志**：能够清晰陈述"要做什么"和"要改什么"。

### 2.2 第二阶段：制定计划

**目标**：将需求拆分为可执行的步骤，列出具体的文件清单。

- 将任务分解为 1 到 N 个独立步骤
- 列出每个步骤要创建、修改或删除的文件路径
- 标注需要更新文档 (`DEVELOPMENT.md`) 的步骤
- 定义每个步骤的完成标准
- 对复杂任务，可使用 `task_plan.md` 等规划文件持久化计划
- **重要**：在获得用户确认后再进入执行阶段

**完成标志**：用户确认计划，或计划已获批准。

### 2.3 第三阶段：执行

**目标**：按计划编写高质量、可维护的代码。

#### 代码修改原则

1. **最小改动、最大兼容**：优先做增量调整而非重构重写
2. **影响范围评估**：修改公共模块（composable、API 层、store、路由、布局组件）时须同时验证所有调用方是否仍正常工作
3. **删除/重命名前全局搜索**：删除或重命名任何导出、路由、CSS 类或 API 字段前，必须全局搜索确认无其他引用
4. **注释原则**：仅标注"为什么这样做"（业务决策、性能优化、边界处理）而非"做了什么"（代码本身可读）

#### UI 相关约定

- 所有交互控件统一使用 `.glass-button` / `.glass-hover` / `.glass-active` CSS 类
- 禁用 `bg-primary-container`、`bg-secondary-fixed`、`text-primary` 等旧样式（作为可点击文字时替换为 `glass-button`）
- Teleport 弹窗：背景遮罩使用 `bg-black/30 backdrop-blur-sm`，弹窗主体使用 `glass-panel rounded-[2rem] bg-white/60` 结构
- 禁用 `bg-black/10`、`bg-black/20`、`bg-black/50` 等其他深浅度的半透明背景
- 弹窗关闭按钮统一使用 `glass-button p-1 rounded` + `close` 图标
- 新页面/新控件不得自创样式模式，必须继承现有的液态玻璃设计系统

#### 数据持久化约定

- 只有真正需要跨会话持久化的业务数据和用户配置才写入 SQLite 数据库
- 临时性数据（如 `last_device_serial`、会话级状态）应使用 `localStorage`
- 新增数据持久化时优先使用 `app_settings` 表的键值对模式（`getSetting`/`setSetting`）
- 结构化数据需添加专用表并在 `migrateInternal()` 中注册 `CREATE TABLE IF NOT EXISTS`
- 新增表后必须同步更新 `AppBackup` / `exportAllData` / `importAllData` / `validateBackup`

#### Rust 后端约定

- ADB 命令避免同时操作同一设备，防止冲突
- 耗时操作使用 `tokio::task::spawn_blocking`
- 外部命令使用 `#[cfg(target_os = "windows")]` 设置 `creation_flags(0x08000000)`

**完成标志**：所有代码修改完成，增量变更已应用。

### 2.4 第四阶段：审查

**目标**：在提交前自我审查，确保质量。

- 逐一检查所有修改的文件，确认没有遗漏
- 检查是否有破坏相邻模块或依赖模块既有功能的变更
- 验证新代码与现有编码风格一致
- 删除调试代码（`console.log`、`dbg!`、临时注释）
- 确认是否需要更新 `DEVELOPMENT.md`（功能变更后必须同步更新）
- 确认修改的功能路径涉及的所有 UI 元素符合液态玻璃设计系统规范
- 如果发现需要重构的地方，记录但不要在当前任务中做——如确需做，需额外提出

**完成标志**：自检清单全部通过，文档已同步。

### 2.5 第五阶段：测试验证

**目标**：确保修改正确且无回归。

必须按以下顺序验证：

1. **类型检查**：`cd test-space && npx vue-tsc --noEmit`
   - 确保无类型错误
2. **生产构建**：`cd test-space && npm run build`
   - 确保构建通过（Vite 构建 + 类型检查）
3. **功能冒烟验证**：对修改涉及的功能路径做人工冒烟（描述验证过程）
   - 确认改动的 UI 交互正常工作
   - 确认 API/Tauri 命令正常响应
   - 确认数据读写正常

**完成标志**：类型检查通过、构建通过、功能验证通过。

## 三、项目编码规范

### TypeScript / Vue

| 规范 | 要求 |
|------|------|
| 组件风格 | Composition API + `<script setup lang="ts">` |
| 命名规范 | 组件 PascalCase，文件 kebab-case，变量/函数 camelCase |
| 类型定义 | 所有 Props/Emits 显式定义类型，禁用 `any`（特殊情况加注释说明） |
| 导入路径 | 使用 `@/` 别名引用 `src/` 下的模块 |
| 响应式 | 优先 `ref()` / `computed()`，复杂状态用 Pinia store |
| 组件通信 | Props 向下传，Emits 向上抛；跨层级用 Pinia / provide/inject |
| 样式 | TailwindCSS 原子类 + 自定义 CSS（仅在需要覆盖组件库样式时） |

### Rust

| 规范 | 要求 |
|------|------|
| Tauri 命令 | 异步操作用 `async fn` + `tokio::spawn_blocking` |
| 错误处理 | 返回 `Result<T, String>`，错误信息用中文或英文（保持统一） |
| 状态管理 | 使用 `tauri::State<T>` + `Mutex` 管理共享状态 |
| 线程安全 | 共享标志使用 `Arc<AtomicBool>` |

### 数据库（SQLite）

- 表名使用 snake_case
- 字段名使用 snake_case，前端查询时用 `AS camelCase` 映射
- 所有表必须有 `created_at`、`updated_at` 字段（如适用）
- 迁移使用 `ALTER TABLE` + `try/catch`（安全重入）
- FTS5 全文搜索作为可选特性，须有 `try/catch` 降级方案

## 四、安全事故处理流程

当检测到可能导致数据丢失、配置损坏或安全漏洞的操作时：

1. **立即停止**并告知用户风险
2. 评估影响范围和严重程度
3. 建议备份（`exportAllData`）后再操作
4. 只有在用户明确确认后（`确认后` / `yes` / `继续`）才执行

### 高风险操作清单

- `DELETE FROM` / `DROP TABLE` 数据库操作
- 删除用户数据或覆盖文件
- 修改路由守卫、认证相关逻辑
- 修改网络安全配置（CSP、CORS 等）
- 执行系统级命令（ADB reboot、remount 等）
- 修改 `tauri.conf.json` 中的安全相关配置

## 五、Agent 行为准则

1. **中文优先**：所有回答、注释、文档输出使用中文（技术术语、关键词、代码片段保持原文）
2. **透明度**：在执行任何可能影响用户的操作前，解释要做什么以及为什么
3. **最小权限**：只操作任务相关文件，不擅自修改未涉及模块
4. **持久化思维**：重要发现、计划、决策写入文件（`findings.md`、`task_plan.md`）
5. **错误透明**：遇错误必须如实报告，不得静默重试或掩盖
6. **文档同步**：功能变更后必须同步更新 `DEVELOPMENT.md` 对应章节
